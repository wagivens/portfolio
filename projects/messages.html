    • Messages Editing Feature

    Chances are, if you have ever used a messaging application, you have sent a message with mistakes and had to resend the corrections with an asterisk at the end (like this*). Because most messaging interfaces do not offer the ability to directly edit text bubbles, users must begrudgingly clutter their threads with re-sent messages instead.

    The absence of an edit feature is particularly shocking for Apple’s Messages application since users can already interact with texts in multiple other ways. This made me curious as to what an edit feature in Messages would look like, as well as how it would work, so I decided to create a potential solution.

    The Rundown

    Summary

    Since this was not a large project, I omitted many steps of the Design Process; I believed doing so would also test my ability to design an effective solution with limited time and resources. As such, my process consisted of: 1. completing a UX audit of the Messages application, 2. sketching my solution and later prototyping it, and 3. receiving feedback from users on the final implementation.

    Timeline and Disciplines

    January 22, 2021 - January 24, 2021

    UX Auditing, Sketching, Prototyping, Figma

    The Solution

    Step 1: UX Audit

    I began by referencing the Messages interface and how users are already able to interact with texts. I wanted to see how Apple could theoretically add this feature without their developers having to create new codebases; as you will see, my solution is similar procedurally to other message interactions, meaning Apple’s engineers would not need to change much of their code to implement this.

    The Messages Application

    The Tapback feature clearly indicates to the recipient that a message has been interacted with (icon in the corner of the text). I considered having an “Edited” icon at the bottom of any message that had been altered.

    Apple also recently added the ability to reply to individual texts. Clicking a reply blurs the interface while focusing on the original message and its reply thread. I thought something similar could be done for edited messages.

    Both features can be accessed by long-pressing a single message, so I believed the edit feature should be accessed the same way.

    Step 2: Sketching

    I then moved on to creating low-fidelity solution sketches. My main goal here was to quickly generate a visual representation of what I envisioned. It was helpful that my ideas were similar to Apple’s existing user interface, so I only had to focus on drawing new features/components.

    Solution Sketches

    I sketched three different screens, each one depicting a different interaction with edited messages: long-pressing to start editing a message, sending an edited message/an edited message thread, and viewing a message's edits.

    Step 3: Components and Prototyping

    Prior to prototyping, I thought about how I could best represent the concept of “editing” without words/through iconography. Ultimately, I decided on a pencil, which can already be found throughout iOS.

    iOS Pencil Icon

    The pencil icon is present throughout iOS and is typically used to indicate that a new instance of something can be made (ex: Mail app, Notes app, Messages app).

    My Pencil Icon

    I designed a pencil icon almost identical to the one found in other places throughout iOS. The icon I created will be inside of a circular container like all of the Tapback reactions.

    Color was important as well. I decided on Apple’s "System Yellow" because Apple typically uses that color when something is being altered or created (ex: in the Photos app while editing pictures, inside of the Notes application, and in the Camera app). In the real world, yellow is often used to denote caution or "work in progress."

    System Yellow

    System Yellow is found throughout Apple's operating systems as indicative of something being edited or created.

    Edited Text Bubble

    Using System Yellow for edited messages would help differentiate message edits from the original text.

    I then put everything together. As aforementioned, the two main actions I focused on were editing a message and viewing message edits. I wanted to make sure both processes would involve similar steps to other message interactions to prevent unnecessarily increasing cognitive load.

    To edit a message, users would simply select the edit feature from the long-press dropdown and send their edits as they would a reply.

    Editing A Message

    Showing the original message while preparing edits prevents users from making any additional errors (by being able to see what was written the first time).

    The yellow "Send" button indicates to users that they are about to send a message edit, not a reply.

    To view any edits made to a message, users would click on the yellow “edited” icon in the bottom left corner of the message (like clicking on a tapback response). They would then see the original message as well as the edits made to it.

    Viewing Message Edits

    Putting the "edited" indicator at the bottom corner of the message separates it from tapback indicators, letting the users know that the message received a different kind of interaction.

    Feedback

    I tested my solution with a few other Messages users and asked for their feedback. Generally, the response was positive, but they specifically liked: 1. using a different color to delineate message types, and 2. preserving the original message to help compare with the edited; keeping the original message could prevent users from concealing/erasing negative evidence (trash talking, explicit photos, etc).

    Reflecting

    I suppose my next step for this would be to conduct proper usability testing with five to six users and consider their feedback before implementing new iterations. I would also optimize for accessibility; some of my design decisions were rooted in the color of components, but many users may not be able to interpret those colors. As such, it would be beneficial to figure out how to convey the same ideas without relying on color.

    Overall, this was a great mini project to exercise my user interface design skills, as well as some fundamental user experience design concepts.
